# Умный гайд по страхованию

Интерактивный веб-проект для кейса I.g.S на IT Purple Hack 2026. Это не витрина страховых продуктов и не калькулятор полисов, а обучающий сервис для подростков 11–18 лет, который помогает понять, где страховка правда может помочь, а где нет.

Проект собран на `Python + Flask`, использует `Jinja` для шаблонов, `vanilla JavaScript` для интерактива и `Docker` для быстрого запуска в контейнере.

## Идея проекта

Подростки редко читают условия страхования как взрослый документ. Им проще разбираться через ситуации из жизни:

- разбил смартфон
- сорвалась поездка
- получил травму на тренировке
- потерял доступ к аккаунту
- не доехал багаж
- не попал на концерт
- упал с самоката
- заметил странное списание с карты
- столкнулся с ущербом от воды

Сервис переводит страховую логику на понятный язык:

- показывает реальные жизненные сценарии
- объясняет базовые термины
- помогает отличать риск от страхового случая
- учит смотреть на документы, сроки и исключения
- не требует регистрации и не собирает лишние данные

## Для кого сделан проект

Основная аудитория:

- подростки `11–13`
- подростки `14–15`
- подростки `16–18`

У проекта есть возрастные режимы. Это не просто визуальное переключение, а разная подача контента:

- `11–13` — максимально простый язык, объяснение через Лёву, знакомые аналогии и бытовые примеры
- `14–15` — более самостоятельная подача, но без тяжёлого формального языка
- `16–18` — ближе к взрослой логике: условия, документы, сроки, ограничения

## Что есть в MVP

### 1. Главная страница

Главная объясняет, что это за сервис, даёт быстрый вход в сценарии и в раздел с базой.

На главной есть:

- вводный hero-блок
- быстрый переход к разбору ситуаций
- быстрый переход в раздел “Основы”
- карточки ключевых сценариев
- короткий блок с базовыми модулями

### 2. Каталог сценариев

В каталоге собраны все жизненные ситуации, через которые пользователь изучает страховую логику.

Сейчас реализованы 9 сценариев:

1. `Разбил смартфон`
2. `Поездка сорвалась`
3. `Травма на тренировке`
4. `Украли доступ к игровому аккаунту`
5. `Потеряли багаж в поездке`
6. `Не попал(а) на концерт или фестиваль`
7. `Упал(а) с самоката или велосипеда`
8. `Списали деньги с карты или кошелька`
9. `Залил(а) комнату или вещи в съёмном жилье`

В каталоге есть:

- поиск
- фильтрация по категориям
- отдельный блок `Сценарий дня`
- адаптация текста под возрастной режим

### 3. Страница сценария

Каждый сценарий — это интерактивный разбор по шагам.

Структура сценария:

- короткое описание ситуации
- блок “что лучше вспомнить”
- пошаговые вопросы с вариантами ответа
- пояснение после выбора ответа
- контекстная подсказка по термину
- итоговый вывод
- памятка: что делать дальше
- мини-квиз для закрепления

Сценарии не обещают выплату и не симулируют работу страховой компании. Они помогают понять саму логику:

- что произошло
- есть ли шанс, что это покрывается
- что может всё испортить
- какие доказательства пригодятся

### 4. Основы страхования

Раздел “Основы” — это набор интерактивных модулей, которые открываются в модальном окне поверх страницы.

Сейчас реализованы 4 модуля:

1. `Что вообще делает страховка`
2. `Риск и страховой случай`
3. `Что делать после события`
4. `Как читать исключения`

Для младшей группы блоки объясняются через Лёву, его истории и очень простые аналогии. Для старших групп подача становится более прикладной.

### 5. Словарь

Словарь объясняет ключевые термины без перегруза.

Сейчас там есть базовые понятия:

- риск
- страховой случай
- покрытие
- исключение
- заявление о случае
- подтверждающие документы
- мошенничество

В режиме `11–13` словарь показывает детские версии объяснений и примеры через Лёву.

### 6. Профиль

Профиль работает в двух режимах:

- без входа — всё хранится локально в браузере
- после входа — прогресс и памятки сохраняются в аккаунте через БД

Там отображаются:

- прогресс по пройденным сценариям
- бейджи
- сохранённые памятки
- выбранный возрастной режим

### 7. Персонализация и доступность

В шапке сайта доступны быстрые настройки:

- крупный шрифт
- тёмная тема
- возрастной режим

Настройки сохраняются локально в браузере.

## Что проект не делает

Важно понимать границы MVP:

- проект не продаёт полисы
- проект не оформляет выплаты

## Как хранятся данные

Проект умеет работать в двух режимах.

### Без аккаунта

- контент — в `Python`-структурах внутри `data.py`
- пользовательские настройки — в `localStorage`
- прогресс сценариев — в `localStorage`
- сохранённые памятки — в `localStorage`

### С аккаунтом

Если пользователь вошёл:

- логин и пароль пользователя хранятся в БД
- пароль хранится в виде хэша
- прогресс синхронизируется с сервером
- памятки синхронизируются с сервером
- настройки и возрастной режим тоже можно восстановить из аккаунта

По умолчанию:

- если `DATABASE_URL` не задан, используется локальный `SQLite`
- если `DATABASE_URL` задан, можно подключить `PostgreSQL`, например на Railway

## Запуск проекта

### Вариант 1. Локальный запуск через Python

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python3 app.py
```

После запуска приложение будет доступно на:

[http://localhost:8000](http://localhost:8000)

### Вариант 2. Запуск через Docker Compose

```bash
docker compose up --build
```

После сборки проект будет доступен на:

[http://localhost:8000](http://localhost:8000)

## Переменные окружения

Для локального запуска можно ничего не задавать, но для прод-режима лучше указать:

- `SECRET_KEY` — секрет для Flask-сессий
- `DATABASE_URL` — строка подключения к БД
- `PORT` — порт приложения в облаке

Пример:

```bash
export SECRET_KEY="super-secret-key"
export DATABASE_URL="postgresql://user:password@host:5432/dbname"
export PORT=8000
```

## Production-конфигурация

Для контейнера используется `gunicorn` с отдельным конфигом:

- [gunicorn.conf.py](/Users/georgijamvrosov/studies/hackathons/IT%20Purple%20Hack%202026/igs/gunicorn.conf.py)

Важно:

- используется `gthread worker`
- это сделано, чтобы избежать нестабильности `sync worker` в контейнере
- порт берётся из переменной окружения `PORT`, поэтому проект можно поднимать не только локально, но и в облаке

## Готовность к проверке по ссылке

Из переписки по кейсу следует, что на проверке очень желательно показать рабочий прототип по ссылке, а не только архив с кодом. В проект уже добавлено всё базовое для такого сценария:

- Docker-сборка
- production-запуск через `gunicorn`
- поддержка переменной окружения `PORT`
- health-check endpoint `/health`
- конфиг для быстрого деплоя на Render
- серверная авторизация и хранение пользовательского состояния через БД

### Health-check

Для внешней проверки и облачного запуска есть endpoint:

- [http://localhost:8000/health](http://localhost:8000/health)

Он возвращает простой JSON со статусом сервиса.

### Быстрый деплой на Render

В корне проекта лежит:

- [render.yaml](/Users/georgijamvrosov/studies/hackathons/IT%20Purple%20Hack%202026/igs/render.yaml)

Как выложить проект по ссылке:

1. Залей проект в GitHub
2. Зайди в Render
3. Создай новый `Blueprint` или `Web Service` из репозитория
4. Render сам увидит `render.yaml`
5. После сборки получишь публичную ссылку на рабочий прототип

Если нужен совсем быстрый путь без ручной настройки, Render здесь удобен именно потому, что проект уже контейнеризирован.

### Railway + PostgreSQL

Если нужен аккаунт и сохранение данных между устройствами, Railway подходит хорошо.

Что нужно сделать:

1. Создать `PostgreSQL` сервис в Railway
2. Создать web-сервис из репозитория проекта
3. Передать в web-сервис:
   - `DATABASE_URL`
   - `SECRET_KEY`
4. Проверить, что healthcheck идёт в `/health`
5. Получить публичный домен Railway

Для этого проекта уже всё подготовлено:

- приложение понимает `DATABASE_URL`
- приложение понимает `PORT`
- есть `/health`
- есть базовая авторизация по логину и паролю
- прогресс и памятки умеют сохраняться в БД
